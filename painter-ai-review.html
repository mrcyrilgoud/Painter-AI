Painter AI

A browser-based paint application with first-class AI affordances — pixel-accurate per-layer canvases, a copilot panel, a ⌘K command bar, and a pluggable backend that can route image generation to OpenAI, Codex, Gemini, or Cursor. This document reviews the current design, the shipped features, and how it's wired together — with two interactive demos at the bottom that exercise the real provider code path.

Visual language

Painter AI mixes a modern editor chrome (Inter + a small Tailwind-style token system) with retro accents — a VT323 wordmark, square pixel-art glyphs in the toolbox, and a chunky 28-color heritage palette. The accent shifts from cobalt in light mode to magenta in dark mode, while AI affordances stay on a consistent lavender (--ai-glow: #b08cff) so they read as a separate layer regardless of theme.

Design tokens

Surface

surface-0 / surface-1 / surface-2 / surface-3 — four progressive lifts, with borders at --border-1 & --border-2.

Type

Inter for the UI, JetBrains Mono for dimensions/keyboard chips, VT323 for the wordmark and a few labels.

Radius & shadow

sm/md/lg = 4/8/12. Two shadows: --shadow-1 for cards, --shadow-pop for menus & the command bar.

AI affordance

A second accent (--ai-glow) ensures AI panels, op proposals, and the ✦ tool stay visually distinct from regular tools.

Theme

Single attribute switch — html[data-theme="dark"] overrides all CSS variables. Persisted via the editor store.

Animation budget

Marching-ants for selection, a 25ms-per-chunk streaming cursor in chat, and fade-up on bubbles. Nothing else moves.

What ships today

Canvas & tools

Per-layer <canvas> stack

  • Each layer owns its own HTMLCanvasElement, stacked absolutely inside a wrap that auto-fits the stage with a 48 px margin.
  • Layers carry visibility, opacity, and blend mode applied via inline style on every render tick.
  • AI-generated layers get an aiProvenance record (mode, prompt, seed, references, mask) used by the History tab.

12 tools

  • Pointer (exits selection mode), Select, Smart Select (point/box/text via /ai/segment), Pencil, Brush, Eraser, Fill, Line, Rect, Ellipse, Text, AI Brush.
  • Pointer events flow through a per-tool handler; strokes commit before/after ImageData to a history stack supporting undo/redo.
  • Keyboard: ⌘Z/⌘⇧Z, [/] for brush size. Losing window focus cancels any in-progress stroke cleanly via a window blur listener.
  • Selection overlays expose 8 interactive resize handles (4 corners + 4 edge midpoints) so the inpaint region can be adjusted after AI output is generated.

AI panel

Chat tab

Streams text + op-proposal events from the Copilot. Selection size, reference count, and canvas dimensions are surfaced as a context chip above the composer.

History tab

Newest-first list of committed AI ops, each with a 48 px thumbnail drawn from the variation's bitmap. Clicking jumps to the associated layer.

References tab

Reference images tagged as style / subject / composition / color with a 0–1 weight slider. Forwarded into every AIGenerateRequest.

Op proposal cards

Inline cards inside the chat thread show the mode, prompt, model confidence, and a 2×2 variation grid. Click to commit; the others fade.

Autonomy selector

Three modes: Propose (default — every op needs a click), Auto-confident (auto-commits the top variation when confidence ≥ threshold), Agentic (multi-step chains).

⌘K command bar

Global, context-aware. The mode chip flips between Inpaint, Outpaint, and New Layer based on whether you have a selection or references. Footer hint reads ↵ generate · ⇧+↵ new line.

Settings tab

Per-session overrides for provider, model, autonomy mode, default style, variation count (1–4), and feather radius — all persisted in localStorage via settingsStore and forwarded into every AIGenerateRequest at fire time.

Floating quick-actions

When a smart-select region is active, four buttons appear above it: Generate, Remove (seamless background fill), Reimagine (variation), Restyle. Each fires a full inpaint request including maskBoundsPx so canvas-code providers can target the edit region exactly. A Re-run button replays the most recent inpaint prompt against the current (possibly resized) selection.

Project & persistence

Autosave

5-second debounced snapshot of the entire project (layers serialized as PNG data URLs) into localStorage. Corrupt blobs are quietly discarded on next boot.

Open / Save / Export

.paintai.json for round-tripping (versioned, currently v1). PNG and JPG export via composite.

Presets

512², 1024², 1024×1536 portrait, 1536×1024 landscape — all wired to File and Image menus.

Status bar

Live dimensions, zoom, and an inline ⌘K trigger. Tool handlers write into setStatusText for transient hints.

Bug fixes · v0.2

  • Fill tool infinite loop + bounds crash — flood fill with a ±1 tolerance matched already-filled pixels and pushed them back onto the stack indefinitely. Fixed with a Uint8Array visited bitset (also eliminates O(4n) stack growth). Coordinate clamping prevents an out-of-bounds startIdx when Math.round lands on the edge pixel.
  • Canvas Size silent data loss — Image › Canvas Size destroyed all layers without confirmation. Now shares the same dirty-check + confirm guard that New Project already used.
  • Smart-select off-by-one — Selection dimensions were maxX − minX / maxY − minY instead of +1, cutting off the rightmost column and bottom row of every smart selection and misaligning subsequent inpaints.
  • Shape overlay ghost — Starting a new shape stroke didn't clear the overlay canvas, leaving the previous drag's preview ghost visible when a prior stroke was interrupted (pointer left window or capture failed).
  • CommandBar keyboard hint — Footer read "⌘+↵ generate" when plain Enter already submits. Corrected to "↵ generate · ⇧+↵ new line".
  • FloatingActions missing maskBoundsPx — Quick-action buttons omitted maskBoundsPx from their inpaint requests, preventing canvas-code providers from targeting the edit region.
  • Window blur handler — Dragging from the canvas out of the window left the drawing state active. A window blur listener in CanvasStage now cancels any in-progress stroke and clears overlay marks left by an interrupted drag.

Architecture

Frontend (Vite + React 18)
  • Zustand stores: editorStore, chatStore, uiStore
  • Per-layer <canvas> elements (no React reconciliation on pixels)
  • Three seams (AIBackend, Segmenter, Copilot) each switchable via Vite env vars
⟶

  • Listens on 127.0.0.1:5174
  • POST /ai/generate, /ai/segment, /ai/chat (SSE)
  • Zod validation + a 32 MB body cap
  • Picks one ImageProvider from IMAGE_MODEL_PROVIDER
⟶
Providers
  • mock — procedural fill
  • openai — gpt-image-1
  • codex-canvas · cursor-canvas · gemini-canvas — model writes a draw() function executed in a sandboxed vm

State management

Four Zustand stores, all instantiated as singletons on globalThis so HMR doesn't fork them:

StoreOwnsNotes
editorStore theme, project name, dimensions, layers (each with its own <canvas>), selection, references, active tool, brush size, primary/secondary color, autonomy, history Mutations to pixel data go through commitPixelChange, which pushes onto a HistoryStack and bumps renderTick to force overlay refresh.
chatStore chat messages: user text, streamed assistant chunks, op-proposal cards, system action-log entries Op proposals own their own state machine: pending → ready → committed/dismissed, plus a progress number while the generator runs.
uiStore command-bar open/closed Kept separate so transient UI doesn't trigger autosave.
settingsStore provider override, model override, autonomy, default style, variation count (1–4), feather radius Persisted via localStorage. Overrides are forwarded into every AIGenerateRequest at fire time, enabling per-session provider/model switching without restarting the server.

The proxy server

The server (server/src/index.ts) is intentionally thin: bind to localhost only, validate every body with zod, cap requests at 32 MB, and hand off to one of five routes. /ai/chat uses SSE so the UI can stream assistant text and an op-proposal event in the same response. /ai/generate queues concurrent requests (default: 1 active, 2 queued) via GenerateQueue so simultaneous sessions don't race the model provider; queue depth is visible at /ai/status.

Image providers

All providers conform to the ImageProvider interface (name, isReady(), generate(req): { variationsBase64, seeds }). The interesting three share a single canvas-code renderer — the model is asked to write a function draw(ctx, width, height), the response is extracted, and the code runs inside node:vm against a Cairo-backed canvas with a 5 s timeout.

ProviderTransportReadiness gate
mock none — server-side procedural fill always ready
openai HTTPS, gpt-image-1 OPENAI_API_KEY set
codex-canvas codex exec --json (subprocess, stdin prompt) codex binary on PATH
gemini-canvas gemini -p <prompt> [-m gemini-2.5-pro] (subprocess) always reports ready — the CLI must be installed
cursor-canvas @cursor/sdk · Agent.prompt(fullPrompt, { apiKey, model }) CURSOR_API_KEY set

Canvas-code rendering pipeline

// shared by codex-canvas, cursor-canvas, gemini-canvas
const raw      = await generateText(userPrompt, DRAW_SYSTEM);         // model call
const code     = extractCode(raw);                                    // strips fences
const canvas   = createCanvas(width, height);                         // node-canvas
await runDrawCodeInWorker(code, ctx, width, height);                  // Worker thread, 5s timeout
const base     = ctx.getImageData(0, 0, width, height);
// 4 variations via fixed R/G/B/brightness shifts on the same base
return { variationsBase64: shifts.map(s => deriveVariation(base, ...s)), seeds };

The trade-off is honest about itself: model latency dominates (one call per request), and variations are cheap recolorings of the same composition rather than independent draws. The win is that the same renderer hosts three very different model transports behind one interface. runDrawCode now executes inside a node:worker_threads Worker (drawCodeWorker.js) so a runaway or slow draw function cannot block the event loop; the 5 s wall-clock timeout is enforced by the Worker's message channel, not a vm timer.

Frontend seams

The three AI capabilities are swappable via Vite env vars at build time:

// src/ai/index.ts
const backendKind   = import.meta.env.VITE_AI_BACKEND   ?? "mock"; // mock | codex
const segmenterKind = import.meta.env.VITE_AI_SEGMENTER ?? "mock"; // mock | codex
const copilotKind   = import.meta.env.VITE_AI_COPILOT   ?? "mock"; // mock | codex

The "codex" implementations talk to the proxy; the proxy then chooses which downstream model serves the request based on its own env. Two layers of indirection — but each is a single dial.

Testing surface

vitest with jsdom — 63 tests across 14 suites. Key coverage areas: the canvas-code renderer (canned draw functions through the real node-canvas + Worker pipeline, asserting on pixel data); the fill tool visited-set and bounds clamping; the generate queue concurrency model; selection bounds arithmetic; inpaint commit compositing; codex client timeout / abort handling; provider readiness; and the proxy's zod validation.

Inpaint demo · drag a region, prompt, replace

    

      This second demo mirrors the app's inpaint mode. The architecture has two halves:
      surrounding pixels become context (the demo samples a palette + 4×4 thumbnail from
      the area around your rectangle and feeds it to the procedural fill as a soft constraint), and
      the rectangle is the only edit window — the result is blended back through a
      feathered mask whose softness you control with the slider. The base outside the rect is preserved
      bit-for-bit.

      gemini-canvas / cursor-canvas /
      openai return real model output. If the server is unreachable, the demo falls back to a
      client-side procedural patch so the flow still works.
